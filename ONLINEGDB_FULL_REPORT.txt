5. SERVER-SIDE ARCHITECTURE (from source code)
================================================================================

5.1 Technology Stack
---------------------
- Node.js (v20.16.0 on server, v24.14.1 locally)
- Express.js on port 3000
- Socket.io for real-time code execution
- Docker (dockerode) for containerized code running
- MySQL (knex) for database
- Redis for caching/queuing
- node-pty for terminal emulation
- New Relic for monitoring
- Prometheus (prom-client) for metrics (/metrics endpoint)
- Winston for logging
- Chokidar for file watching

5.2 Code Execution Flow
------------------------
1. User writes code in Ace Editor
2. Socket.io emits 'run' event with {src, lang, stdin, cmd_line_args}
3. Server receives via socket.on('run', ...)
4. Code written to temp directory
5. Docker container spawned with user code mounted
6. Output streamed back via socket.stdout / socket.stderr
7. Container cleaned up after execution

5.3 Docker Images
------------------
  ogdb/pack:base           (Ubuntu Trusty - 14.04)
  ogdb/pack:focal          (Ubuntu 20.04)
  ogdb/pack:jammy          (Ubuntu 22.04)
  ogdb/pack:noble          (Ubuntu 24.04)
  ogdb/pack:noble-minimal  (Ubuntu 24.04 minimal)

  Start: SHELL_BASED_EXECUTION=1 SANDBOX_DOCKER_UBUNTU_FOCAL=jammy
         node --tls-min-v1.0 main/app full_cpu
  Port:  3000

5.5 Security Weaknesses Found in Code
--------------------------------------
- execSync, exec, spawn used with user-controlled input
- No authorization on socket.io connections
- Docker socket exposed (/var/run/docker.sock)
- MySQL credentials stored in plaintext in config files
- SSL private key in source repository
- No rate limiting on compilation requests
- User code runs without proper seccomp/apparmor profiles
- New Relic API keys hardcoded in client-side JS
